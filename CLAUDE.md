# CLAUDE.md

Guidance for working in this repository — the distilled, load-bearing version. The full design narrative (the *why*) is in [dev_ref/rough_design.md](dev_ref/rough_design.md); the detailed, living phase plan + status is in [dev_ref/roadmap.md](dev_ref/roadmap.md) (the source of truth for what's done / next).

## What this is

An R **Shiny** dashboard that lets people with little to no bioinformatics background **explore, manipulate, and visualize RNA-seq results**. The input is an R object — a `DESeqDataSet` (`dds`) or a `SingleCellExperiment` (`sce`) — or a **counts matrix + sample sheet** (CSV/TSV/XLSX) that the app assembles into a `dds`.

**Scope is bulk-first** (decided in review). Build the bulk path end-to-end first; keep the design general enough to add single-cell later. An `sce` is supported via two routes: **pseudobulk** aggregation by a sample grouping, or **per-cell coercion to `dds`** behind a *"statistically inaccurate and slow"* warning. **Force pseudobulk above ~1k cells.** (Why per-cell DESeq2 is unsound: [dev_ref/normalization-scran-vs-deseq.md](dev_ref/normalization-scran-vs-deseq.md).)

The minimum viable input is a `dds` carrying only a raw count matrix; everything else (sample metadata, gene metadata, feature lengths, normalized assays, DE results) can be added inside the app.

## Tech stack

- **R + Shiny**, UI built with **bslib** (Bootstrap 5). Use the `shiny-bslib` and `shiny-bslib-theming` skills for layout/theming. `page_navbar()` for the multi-page structure; `DT` for editable tables; `shinycssloaders` for spinners; `thematic` so ggplots match the theme.
- **Bioconductor:** `DESeq2`, `SummarizedExperiment`, `SingleCellExperiment`, `scater` (QC), `rtracklayer` (GTF), `edgeR` (`filterByExpr`), `apeglm`/`ashr` (LFC shrinkage), `AnnotationDbi` + `org.*.eg.db` (annotation), `scran` (single-cell, later).
- **Plots:** `ggplot2` is the default for all plots. `ComplexHeatmap` is the heatmap engine (do not substitute pheatmap/heatmaply).
- **Tables / IO:** `readr`/`readxl` in, `writexl` out; `rmarkdown` + `sessioninfo` for the reproducibility report.

Domain conventions (object shape, assays, normalization math, QC metrics, DE result columns, `feature_class`/`feature_length`) live in the **`rnaseq-bioc`** skill — consult it before touching data logic.

## Project layout

This is a **plain R package** named `ddsdashboard` (not golem). Reusable helpers live in `R/` and are exported so other lab projects can `library(ddsdashboard)` and call them; the Shiny app is just `run_app()`.

```
DESCRIPTION / NAMESPACE   # package metadata + exports (regenerate NAMESPACE with devtools::document())
environment.yml           # conda/mamba env (source of truth for deps)
R/
  run_app.R               # run_app(), app_ui(), app_server() — the navbar shell + module wiring
  state.R                 # new_app_state() + the state_*() helper API (the canonical store)
  mod_<page>.R            # one Shiny module per page (input, metadata, feature, assay, qc, dimreduc, de, geneset, expression, export)
  mod_<shared>.R          # reusable sub-modules embedded in pages (mod_meta_editor, mod_gtf_reader, mod_statusbar, mod_design_builder)
  mod_plot_engine.R       # plot_engine_server(): the ggplot↔plotly dual_plot() + deferred()/stale_note() render gate
  mod_plot_subset.R       # plot_subset_*(): the view-only "Plot Showing" sample-subset control
  mod_expr_value.R        # expr_value_*(): shared assay/transform/pseudocount control (DE Direct plot now; P7 Expression)
  mod_continuous_palette.R # continuous_palette_*(): reusable colour-ramp control (base palette + anchors + reverse + custom stops), extracted from the Palette page (P7c heatmap; P8 shared heatmap controller)
  aes_helpers.R           # the shared colour/annotation attribute catalog + resolver (see shiny-plot-aesthetics)
  <topic>_helpers.R       # pure, exported, tested helpers (annotation_, gtf_, metadata_, assay_, load_)
  utils_normalization.R   # cpm(), tpm(), fpkm(), logcounts_from_counts() — pure, dependency-free, tested
  utils_lookup.R          # lookup_feature()/resolve_feature()/suggest_features() — feature id <-> name resolution
  utils_group_choices.R   # group_field_choices() — grouped colour/group-by selectInput choices
  utils_table.R           # dt_table() — the standard read-only DT wrapper (filters + rows-per-page)
  mock_data.R             # make_mock_dds() etc. — fixtures used by tests + the demo
data-raw/                 # scripts that build mock dds/sce fixtures for tests + demo
dev_ref/                  # developer reference notes (rough_design.md, roadmap.md, normalization-scran-vs-deseq.md, object-states-and-resets.md) — all build-ignored
tests/testthat/           # unit tests for helpers + module servers (shiny::testServer)
inst/extdata/             # bundled reference data (ERCC Mix1/Mix2 concentrations: ercc_concentrations.csv, see ERCC_SOURCE.md)
```

The file conventions: **`mod_<page>.R`** = a navbar page; **`mod_<shared>.R`** = a reusable sub-module embedded inside pages (e.g. the draft metadata editor, the GTF reader); **`<topic>_helpers.R`** = pure, Shiny-free, exported functions that the modules call (so they stay reusable and unit-testable).

Module contract and the shared-`state` wiring are documented in the **`shiny-module`** skill (which also covers the reusable sub-module + draft-editor patterns, the `dual_plot` engine, `mod_plot_subset`, and deferred render). New pages follow that pattern. **Any plot that colours/groups/annotates by a per-sample attribute uses the shared catalog + resolver (`aes_helpers.R`) — see the `shiny-plot-aesthetics` skill** (the canonical reference for the attribute model + palette standard). Domain/statistical conventions (object shape, assays, normalization, QC, DE schema, `feature_class`/`feature_length`, ERCC) live in **`rnaseq-bioc`**; the **`annotation`** skill covers the OrgDb + GTF row-annotation write conventions (consult it before touching mapping/annotation logic). A **`de-analysis`** skill (design/contrast/shrinkage) may be added when Phase 5 lands. The **`bioc-reviewer`** agent reviews any data-logic change; the **`shiny-feature-builder`** agent implements a page/helper end-to-end.

**Dependency policy:** `DESCRIPTION` `Imports` holds the always-on framework deps the app loads unconditionally — currently `AnnotationDbi`, `bslib`, `DESeq2`, `DT`, `methods`, `readr`, `readxl`, `S4Vectors`, `shiny`, `SummarizedExperiment`, `thematic`. (`S4Vectors` is Imports because the size-factor config uses its `metadata()`/`metadata<-` accessors unconditionally — SummarizedExperiment re-exports the setter but not the getter, so both are `@importFrom`'d and called bare.) Heavy or optional packages are loaded **on demand via `requireNamespace()`** (with graceful degradation / a `skip_if_not_installed()` in tests) and stay in `Suggests` *even though code calls them* — currently `ComplexHeatmap`, `GenomicRanges`, `IRanges`, `rtracklayer`, `org.*.eg.db`, `ps`, `scater`, `SingleCellExperiment` (and, planned for the P7 Expression single-gene plot, `ggbeeswarm` — `r-ggbeeswarm` on conda-forge — with a `geom_jitter` fallback). They're also pinned in `environment.yml`. Promote `Suggests → Imports` only when a feature loads the package unconditionally. The pure base-R utilities (`utils_normalization.R`) deliberately take no dependencies so they stay reusable and fast to test.

## Environment & running

Dependencies are managed with **mamba** (channels: `conda-forge`, `bioconda` only — never `defaults`/`anaconda`). The environment is `rnaseq_dashboard`.

```bash
mamba env create -f environment.yml      # first time
mamba env update -f environment.yml --prune   # after editing deps
conda activate rnaseq_dashboard
```

Dev loop (from the repo root, env active):

```r
devtools::load_all()      # source the package
run_app()                 # launch the dashboard
devtools::document()      # regenerate NAMESPACE/man from roxygen
devtools::test()          # run testthat
devtools::check()         # full R CMD check
```

Clone-and-run for end users: create the env, then `R -e 'devtools::load_all(); run_app()'` (or install once with `devtools::install_local(".")` and `library(ddsdashboard); run_app()`).

## Architecture

Multi-page dashboard. Each page is a **Shiny module** (`mod_<page>_ui()` / `mod_<page>_server()`) so pages stay isolated and testable. Build a new page with the **`shiny-module`** skill.

### State model (the load-bearing design)

App state is one `reactiveValues` (`new_app_state()` in `R/state.R`) interacted with through the `state_*()` helper API (`state_load/mutate/derive/reset/undo`, `state_dds`, `state_meta`):

- **`original`** — the object exactly as loaded; never mutated; the reset target.
- **`working`** — the current `dds` after edits/filters/normalization (always a `dds`, even for `sce` input via pseudobulk/coercion).
- **`data_version`** — bumped on **any** edit to samples, features, or design-relevant metadata.
- **`history`** — an ordered action log (one entry per operation, with parameters); renders the reproducibility export.
- **`undo_stack`** — a short snapshot stack (depth 5) of prior `working` objects for `state_undo()`.
- **`meta`** — app-level flags not stored on the `dds` itself: `data_type` (bulk/single-cell), `feature_type` (drives `<feature_type>_name` labels and the status bar), `sce_per_cell`.
- **`derived`** — cached heavy artifacts (size factors, VST, PCA, DESeq fit, DE result tables), keyed by name. **A plain environment, not a reactive field** (writing cache entries must not trigger reactivity); staleness is keyed on `data_version` via `state_derive()`. **Exception — content-addressed artifacts** (`.content_derived_keys` = `de_fit`, `vst`): these depend on the count matrix, sample/feature sets, and **size factors** but *not* on which derived assays are present, so they key on `dds_content_fingerprint(dds)` (`rownames`+`colnames`+`sizeFactors`) instead of `data_version` — pass it as `state_derive(..., version=)`. `state_mutate` **preserves** these across its `.clear_derived(keep=.content_derived_keys)` (they self-invalidate by fingerprint on read), so adding a normalized assay (CPM/TPM/FPKM) no longer wipes/stales the DESeq fit or VST, while a real sample/feature drop or a size-factor change still does. Reset/undo clear everything.
- **UI-preference / session fields** (reactive, but **not** data — untouched by `data_version`): `plot_interactive` (the ggplot↔plotly engine toggle), `palette` (the Palette page configs), and `samp_pool` / `samp_flags` — the QC removal pool + latest `flag_samples()` frame, **promoted from the QC page** (`mod_qc` proxies the pool and mirrors the flags) so other plot pages can colour/annotate by removal/pool. Cleared on data load.

**Invalidation (conservative v1):** each `derived` artifact records the `data_version` it was computed under and is **stale** when that differs from the current `data_version` (except the content-addressed `de_fit`/`vst`, which key on `dds_content_fingerprint()` — see `derived` above). The UI shows a stale badge + re-run button and **never silently reuses stale results**. Dependency order: `raw counts → filter → normalized assays → variable genes → PCA → DESeq fit → DE results → heatmap`. **Reset-to-original** restores `working <- original` + clears `derived`; **undo** keeps a short snapshot stack. These two are surfaced as **global Undo / Reset buttons in the status bar** (`state_undo()` / `state_reset()`); every committed edit (metadata Save, assay add, sample/feature removal) is undoable. **Scoped, local resets** complement them: the metadata editor's "Reset to original" reverts just that slot (`colData`/`rowData`) and **auto-commits** (no Save; it also shows an "unsaved changes" badge by Save), and the QC Filtering pills' "Reset Sample/Feature Removal" re-add removed items from `original` while keeping other edits (`reset_metadata_slot()`, `restore_samples()`/`restore_features()`). See [dev_ref/object-states-and-resets.md](dev_ref/object-states-and-resets.md). A **persistent dataset-status bar** on every page surfaces data type, n samples/features, assays, design-set?, and DE current/stale. The `history` log renders the **reproducibility export** (R script / Quarto report). Modules still follow the `dds`-in/`dds`-out contract (see `shiny-module`), but the canonical store is this state object.

Pages (navbar order). The first navbar entry, **Input**, is itself a sub-tabbed group (`navset_card_tab`) holding Dataset / Sample / Feature / Assay; the rest are top-level pages, plus a persistent status-bar `nav_item`:

1. **Input** (sub-tabbed group):
   - **Dataset** — load `dds`/`sce` **or** counts matrix + sample sheet; show `show(dds)`.
   - **Sample** — a `navset_card_pill` of *Sample Metadata* (edit `colData` via editable `DT`) and *Additional Metadata* (upload + bind a sample sheet). Built on the shared draft editor (`mod_meta_editor`).
   - **Feature** — a `navset_card_pill` of *Feature Metadata* (edit `rowData`, set `feature_class`/feature unit), *OrgDb Annotation*, and *GTF Annotation*. Annotation (OrgDb organism-first, then GTF which overrides) composes onto the editor draft and commits on Save; detect/confirm feature type; tag exogenous / ERCC spike-in. See the **`annotation`** skill.
   - **Assay** — add normalized assays (CPM/TPM/FPKM); store `feature_length`. **Assays only** — size factors are decoupled (they're a DESeq2 median-of-ratios normalization, unrelated to these library-size/length assays) and live on the next sub-tab.
   - **Size factors** (`mod_sizefactors.R`) — the deliberate DESeq2 size-factor normalization, a **`navset_card_pill` of Estimate / Per-sample / Compare** (Compare last). **Estimate**: pick what to **estimate using** (radio, label "Estimate using:") — endogenous default / spike-in / a **custom set via the shared `mod_gene_search`** / **all genes (discouraged, last option)** — + the `estimateSizeFactors` **`type`** (ratio/poscounts/iterate), then estimate; shows the per-sample factors + provenance. **All types honor the control set** by estimating on the control-gene **row-subset** and having the full dds **inherit** the per-sample factors (identical to `controlGenes=` for ratio/poscounts; makes `iterate`, which ignores `controlGenes`, respect it too; `iterate` is refused when the control set is too small to identify the design). An empty spike-in/custom control set is a clear error (never a silent widen to all genes); a **gene-set-type-specific warning** for each non-endogenous choice (spike-in noise / a custom set's stability / `all_genes` ignoring the spike-in/exogenous split) notes it drives DE/PCA/Expression, not just QC. Estimation shows a `withProgress` busy indicator (some `type`s, e.g. `iterate`, are slow). The config is carried **on the dds** (`metadata(dds)$sizefactor_config` via `sizefactor_config()`/`set_sizefactor_config()`) so it survives edits and **structural re-estimations reuse it** (`reestimate_size_factors()`). **Materialized by default at load** (`ensure_size_factors()`: respects an object's own factors as provenance `"loaded"`, else the endogenous default as `"auto"`) so DE/PCA/Expression read one consistent, visible normalization; those pages are **consumers** — `de_run()` never writes size factors back. Estimation is **idempotent** (re-estimating the same config is a no-op, no `data_version` bump). `estimate_size_factors(dds, config)` is the generalized helper; `estimate_size_factors_endogenous()` is the thin wrapper kept for internal callers. **Per-sample**: the current size factors as a plot — a per-sample bar (with a Sort-by control) when x = Sample, or points grouped by a `colData` var (point size/opacity shown only then); colour defaults to the design's variable of interest (`primary_design_var()`); shared `dual_plot`/deferred engine + `mod_plot_subset` "Showing:" + `aes_helpers` colour. **Compare** (consumer-only): a scatter of **two** size-factor vectors computed under two different configs (two independent method accordions, each with its own `mod_gene_search`) with an **x=y dashed line + linear fit R²** to gauge similarity (e.g. spike-in vs endogenous); it calls `estimate_size_factors()` twice **read-only** (pulls `sizeFactors()`, **never mutates `state`** — no `data_version` bump, nothing saved to the dds; it never reads the dds's *committed* size factors either). Cached in `derived` and gated on a **content token** (rownames/colnames/`feature_class`/`design_version`) that **deliberately excludes the committed `sizeFactors(dds)`** — so an Estimate-pill size-factor change neither re-estimates nor stales the Compare plot (its stale banner would otherwise misfire), while a real structural edit (sample/feature/reclassification/design) does re-estimate. **No auto-render** (re-estimation can be slow → a manual **"Compare"** button + `withProgress`); plot-setting changes (colour/size/height) re-plot live from the cached vectors, and an **adjustable plot height** (`dual_plot` accepts a reactive height) keeps the fixed-ratio square from being pinned small. Size factor is also a plottable **General QC metric** (`qc_per_sample_metrics()` `size_factor` column → `.qc_metric_choices`).
2. **QC & filtering** — one **unified** page (a `navset_card_tab`) carrying a **data-type badge**. Sub-tabs: **Dataset diagnostics** — the VST mean–SD plot (rank-mean vs sd, points coloured by local density à la `geom_pointdensity` but computed with base `MASS::kde2d` to avoid an extra dep; VST via `qc_vst()`, which falls back from `DESeq2::vst()` to `varianceStabilizingTransformation()` on small/low-count data). **Sample QC** (pills): *General QC* (per-sample metrics — library size, detected, % mito, % spike-in; `qc_helpers.R`, `scater::perCellQCMetrics` + base-R fallback), *RLE*, *Expression density*, *QC Matrix* (the `dt_table`); each ggplot pill has its own group/colour control. **Sample Correlation** (pills): *Heatmap* (`ComplexHeatmap`, log-counts, Spearman/Pearson shown in legend+title, multi-column annotation with a Clear-annotation button) and *Within-group correlation* (`qc_within_group_correlation()` — per-sample mean correlation to same-group samples; candidate-outlier read). **Spike-in (ERCC)** (`ercc_helpers.R`) — **titration QC** (*not* normalization; size factors stay endogenous-`controlGenes`): a *Dose-response* log–log scatter of known concentration vs observed expression with a per-sample fit, and a *Per-sample summary* (% spike-in, detected-spike count, slope, R², lowest detected conc + spike-fraction CV). Concentration source is selectable — a `spike_concentration` rowData column (when present) or the bundled **ERCC Mix 1/2** (`ercc_concentrations()`, join by id); observed assay is a linear depth-normalized one (**TPM > FPKM > CPM**-preferred since length-normalized abundance tracks molar spike-in input; never counts/logcounts); zeros dropped before the log, fit NA-safe below 3 points (`spike_dose_response()`). All sample diagnostics compute on **endogenous** features only. **Diagnostic plots carry a short "How to read this" guideline below them** (`.qc_diag_help`). Heavy outputs show `shinycssloaders` spinners; VST/correlation/load show `withProgress`. **Filtering** (pills *Samples* / *Features*, `filter_helpers.R`): the app *suggests* low-quality items with per-reason detail (`flag_samples()` — opt-in library-size/detected/%-mito + within-group-correlation-outlier checks, each **blank threshold = disabled**, with an **Auto** button that fills robust median±3·MAD fences via `suggest_sample_thresholds()`; `flag_features()` — all-zero + `edgeR::filterByExpr()` or a manual `rowSums`/detected rule); **expression filters run on `endogenous` only — `spike_in`/`exogenous` exempt** (consistent with `controlGenes`); **avoid `HTSFilter`**. The user builds a **removal pool** (read-only *Suggested* column + separate *In pool* column; DT row-selection stages, buttons commit — Add/Remove selected, **Add all in current view**, **Adopt suggestions** as a union, Clear) and **Applies** it via `state_mutate` (a real, undoable removal → `drop_features()`/`drop_samples()`, which `refresh_assays()` and re-estimate size factors). **Features pre-seed the pool with the suggestion; samples are highlight-only / opt-in** (rnaseq-bioc: small-n outliers are flagged, not auto-dropped). The Features pill shows the **before/after log-CPM filtering density** (`qc_filter_density()`); Sample QC plots offer a **"Removal status" colour-by** (reason-aware green/yellow/red via `removal_status()`). A **"Showing:"** control lives in every sample-plot sidebar (Show-by column → *Keep* values, blank = show all) — all instances stay **synced** to one canonical selection (`show_by_rv`/`show_values_rv`); it hides samples from the sample plots + correlation heatmap **without** mutating the `dds` (view-only — diagnostics still compute on all samples; no `data_version` bump) and re-renders only on the tab's Render/auto (folded into each deferred spec). DT selected-row highlight uses the theme primary colour app-wide (`inst/www/custom.scss`).
3. **Dimensionality reduction** — **PCA** (`mod_dimreduc.R` + `dimreduc_helpers.R`; t-SNE/UMAP deferred to P4c). Input matrix via `pca_input()` — **VST by default** (`assay(qc_vst())`, broad fallback to `logcounts`), or `norm_logcounts` / a stored assay; `pca_assay_advice()` tiers inputs (recommended / log-first CPM-TPM-FPKM / unsuitable raw counts) and drives a warning + a default `log2(x+1)` toggle. Top-variable genes (`top_variable_features()`, default 500, **endogenous only**) → `compute_pca()` (`prcomp(t(top), center=TRUE, scale.=FALSE)`, %var from `sdev^2`, deterministic PC sign). The embedding is cached via `state_derive` (keyed on assay/n_top/log — **not** the shown subset) behind the deferred Render gate; the scatter has **PC-axis selectors**, colour by metadata (discrete/continuous **Palette** configs) or gene expression (logcounts gradient via `lookup_feature()`), shape by discrete metadata (NA→"NA"), an **input-used subtitle**, and a **scree** %-variance bar. Reuses the shared plot engine (`mod_plot_engine.R`: `dual_plot`/`deferred`) and the "Showing" subset (display-only — embedding/axes stay stable). Multi-panel (1/2/4) is P4b.
4. **Differential expression** (`mod_de.R` + `de_helpers.R` + shared `mod_design_builder.R`) — a `navset_card_tab` of **Design & Contrasts** / **DE Plots** / **Results Table**. *Design & Contrasts*: the shared design builder (synced with the Dataset **Design** sub-tab; guided design + reference level + full-rank check) + contrast picker (three validity tiers extractable/not-in-design/invalid) + **Run DESeq2** (the fit is contrast-free, stored in `derived` under a `(data_version, design_version)` stamp) + **reactive/auto result extraction** (per-contrast cached in `state$de$results`, apeglm→ashr→normal shrinkage). Results carry **both** `log2FoldChange`/`log2FoldChange_shrunk` + matching `sig`/`DEG`/`sig_shrunk`/`DEG_shrunk` (shrinkage toggle = column selection). *DE Plots*: **MA / volcano / direct-comparison** on the shared `dual_plot` engine (segmented plot-type over one deferred plot; render controls above the plot; **no `mod_plot_subset`** — per-feature), colour by DEG (from the Palette **`other/DEG`** set — which also offers the generic discrete palettes) / None / a DE column via the **local** `de_colour_resolve` (the per-sample `aes_helpers` resolver doesn't apply), **field-based axis limits** that **set real coord limits** (extend the axis when wider than the data) → triangle clamping, DEG points drawn **`no_change`-first** (DEGs on top), high-contrast ggrepel labels, `fixed_ratio` 1:1 toggle on Direct; the Direct plot's expression value (assay/transform/pseudocount) comes from the shared **`expr_value_*`** control (`mod_expr_value.R`, reused by P7). *Results Table*: `dt_table()` with DEG colouring + significant-only. A shared **"Contrast & thresholds"** group (contrast-to-view + padj/|LFC|/use-shrunk) sits in **all three tabs**' sidebars, synced via `state$de$active` + a canonical `thr` store; the Run-note reflects **fit** status (`de_fit_status`), the DEG summary reflects **results** status (`de_status`). Design changes are **design-scoped** (`state_set_design`: bump `design_version`, not `data_version`, so PCA/VST/QC caches survive). See the **`shiny-de-analysis`** skill.
5. **Gene Sets** (`mod_geneset.R` + `gene_set_helpers.R`) — define/record/manage **named gene sets of interest** (DE seeds them; the P7 Expression heatmap consumes them). A **staging** Manage tab: a *Build a gene set* card (a **source pill** → a live **Preview** of the *staged* genes → a **New / Add-to-existing Save**, New rejecting a name clash) beside a *Your gene sets* card. `staged()` is a **named list** of incoming id-vectors following the active source — one source stages one set (**Paste** via the shared `mod_gene_search`; **From DE** DEGs with the page's own padj/|LFC|/shrunk + a stale-fit warning; **Top-variable** via `top_variable_features()`), while the **table import** (CSV/TSV/XLSX: `dt_table` view/filter/select, ID-column + match-field pick with auto-detect, annotation-split) can stage **many** (→ a multi-set Save with a name-conflict resolver). The import's name→id matching is **1:many keep-all** (a set is a collection; toggleable to first-only) via its own resolver — *not* `lookup_feature()` (see the Feature-lookup ID-match convention below). Stored in **`state$gene_sets`** — a named list of **structured records** `list(ids, kind, annotation, source)` (`kind`/`annotation` forward-compat with the P7 annotated layer); a **session/UI field** (no `data_version` bump, cleared on load). **Non-destructive:** `ids` is the full authored membership, never trimmed; "present/absent in the dataset" is a live derived view (`gene_set_present()`/`gene_set_absent()`), and a data edit only *notifies* about newly-absent ids (never mutates the store). **File round-trip** (P6d): `gene_sets_to_json`/`_gmt`/`_tsv` + inverses + a `gene_sets_from_file()` extension/sniff dispatcher — **JSON is the faithful mirror** (round-trips `source`/`kind`; a 1-gene set stays an array via `I()`), **GMT** carries `source` in its description field, the **long TSV** is `set`/`id`(/`annotation`); import is a **"Gene set file" source pill** whose named sets stage through the existing **multi-set Save** (New / Add-to-existing — non-destructive, never a whole-store Replace), reusing the table-import **ID-match scheme** (match against rownames or a rowData column with auto-detect, 1:many keep-all/first, and a keep-unmatched escape hatch) so a file authored in another id scheme (e.g. `gene_name`) resolves. Export is its **own section** (separate from *Your gene sets*): a **selective** set multi-selector (Select all / Deselect all) + format selectize + a live capped **Preview** of exactly what downloads (mirrors the Palette Config tab). The **Compare tab** (P6e) follows the DE-Plots layout — a `bslib::layout_sidebar` whose **sidebar** carries the two shared controls governing both sub-pills (a **"Sets to visualize"** multiselect via the shared `.gs_set_multiselect_ui`/`_server` helper, also used by Export; and a **"Within this dataset only"** toggle, `gene_set_present` vs full authored `ids`), over a `navset_card_pill` **Stats** | **Overlap** whose per-pill controls sit above each plot. *Stats* = a present/absent set-size bar on the shared `dual_plot` engine (horizontal default + a Vertical toggle; an **Order by** control — None / increasing / decreasing / name, via `gene_set_size_frame(order=)`; **present** stacked nearest the axis via `position_stack(reverse=TRUE)`; colours from a static **`other/gene_set_presence`** Palette item resolved by `gene_set_presence_colors()`, mirroring `removal_status_colors()`). *Overlap* = **Euler / Venn / UpSet** where the **"Diagram type" default follows the set count** (`.gs_first_suitable_type()`: Euler ≤3, Venn 4, UpSet ≥5) until the user picks a type; a selected type that can't draw the current count shows a **message and no plot** (`.gs_type_valid()`) — never a silent substitution. Pure, tested Compare helpers: `gene_set_size_frame()` / `gene_set_overlap_list()` / `gene_set_ids_for()`. **Overlap dependency:** `eulerr` (area-proportional Euler + Venn) has **no conda-forge osx-arm64 build** and `eulerr >= 7` needs a Rust toolchain, so it is a **CRAN install** — the one documented exception to the conda-only policy (see `environment.yml`); the conda-clean **`ggVennDiagram`** is the Venn fallback when eulerr is absent (the Euler option is then hidden), and UpSet uses `ComplexHeatmap::make_comb_mat()` (both non-ggplot → static `renderPlot`, `requireNamespace()`-guarded). **Deferred to P7:** the annotated layer (id→label sets + a combine-sets-into-annotation resolver for the heatmap's row annotation) + a `Palette > Gene Set` per-set-colour sub-tab. Recorded in the reproducibility export.
6. **Expression** (`mod_expression.R` + `expr_helpers.R`) — exploring expression of genes of interest, in more than one form. A `navset_card_tab` with two tabs (**Single genes**, and **Gene sets** — itself a `navset_card_pill` of *Aggregate expression* + *Heatmap*). **Single genes** — one feature at a time as a layered overlay (back→front: violin → boxplot → dots), reusing the shared plot machinery (`dual_plot` engine, deferred render, `mod_plot_subset` "Showing:", `aes_helpers` colour). x-axis = a metadata grouping var (default first design var; `group_field_choices()`, **colData-only**); y-axis value via the shared **`expr_value`** control — extended with opt-in **VST** + **normalized log-counts** synthetic choices (backward-compatible; DE's use unchanged) so **any stored assay or VST/norm-log-counts is selectable**; default via `expr_default_assay()` = **`norm_logcounts` when size factors exist, else `vst`** (both resolved by `expr_value_matrix()`, VST endogenous-only via `qc_vst()` with a logcounts fallback). The value **matrix** (incl. VST) is cached in `derived` (keyed on assay + `data_version`); the **source** (gene / gene set) + value + transform + aggregation sit **behind the Render gate** (the deferred `sig` includes the source, so with auto-render off a gene/set change waits for Render), while grouping, colour, and geom toggles are **live re-plots** on top of the gated result. (`.expr_dist_server()` shares this machinery between both distribution pills; a `reduce` fn returns `list(ok, ...)` rather than `validate()`-ing, since a deferred spec swallows `validate()`.) group/colour-by independent (`aes_resolve()`; discrete → violin/box fill + dot colour, continuous → dot colour); transform+pseudocount (`expr_transform()`); a **dot-layout selector** (`ggbeeswarm::geom_beeswarm` cex / `geom_quasirandom` width / `geom_jitter` width — jitter is also the **fallback** when `ggbeeswarm` is absent) + per-layer width/opacity/size controls + a **y-axis limit** control (min/max; out-of-range dots draw as boundary triangles via `de_clamp()` + `coord_cartesian()` — the DE-plot pattern; distributions are zoom-clipped, not distorted); static Render controls above the accordions. **Sample guards** (`expr_geom_availability`; `option()`s `expr_dist_min`/`expr_dots_max`/`expr_dots_hard` = 10/100/500): violin/box shown once any group `N ≥ dist_min`; dots default ON when `N < dots_max`, disallowed when `N ≥ dots_hard`. **Gene sets > Aggregate expression** — the SAME layered overlay, but the y value is a per-sample **gene-set score** (`expr_set_aggregate()`): drop absent + optionally all-zero-count genes (**Only genes with expression**, default on), transform, **per-gene z-score by default** (so highly-expressed genes don't dominate — comparable across TPM/VST), then **mean/median** down to one vector. Source = a **saved set** or a **quick uncommitted search** (`mod_gene_search`, multi). Default value = **TPM when `feature_length` exists, else VST**. Subtitle always reports **"<Mean/Median> expression of n of N genes (x%) within the set"** (+ a non-varying note); no plot when 0 survive. The single-gene and aggregate pills share grouping/colour/geoms/styling/deferred-matrix via **`.expr_dist_server()`** (one reduce fn differs). **Gene sets > Heatmap** — `ComplexHeatmap` over a named set (**blank until a set is chosen** — no DEG/top-variable auto-default). **Render-only** (a heatmap is slow; a single static `renderPlot`+`draw()` has **no cheap live layer**, so *every* control is snapshotted on Render and a settings change shows a stale banner — unlike the `dual_plot` pages' snapshot/live split) + a spinner. Value matrix via the shared **`expr_value`** control (any stored assay / VST / norm-log-counts), cached in `derived` behind the gate; **default per-gene z-score** (toggle raw `log10(assay + pc)`) — constant/zero-variance rows z-score to 0 (never `NaN`, via `row_zscore()`) so they draw flat instead of breaking the heatmap, and are counted. **Row/column display labels are decoupled from the matrix dimnames** (`expr_heatmap_matrix()` keeps the unique dds ids as the matrix `dimnames` — the stable key for subsetting / `anno_mark` indexing / clustering — while ComplexHeatmap's `row_labels=`/`column_labels=` get a display vector from `expr_heatmap_labels()`): so duplicate `gene_name`s (**many ids → one symbol**, an intrinsic annotation property) and NA labels (→ id fallback) are safe, never a duplicate-`dimnames` error. Label modes **Auto / All / Selected / None** (Auto = `heatmap_label_default(n, max)` by size — **separate row/column thresholds**, `option()`s `heatmap_row_label_max`/`heatmap_col_label_max`/`heatmap_dend_max`); **Selected** marks searched genes/samples via `anno_mark()` and reports how many can't be shown (`expr_label_coverage()` — searched ids from the whole dds, not just the plotted rows). Sample (top) annotation via **`aes_annotation()`**. **Deliberate exception to the "display aesthetics stay live" rule:** a ComplexHeatmap redraw is *not* a cheap re-plot, so **every** aesthetic here — the value ramp *and* the annotation colours — is **gated behind Render** (resolved into the snapshot, with `hm_ramp()` + `input$hm_ramp_src` + `state$palette` in the deferred `sig`), so a colour/Palette edit shows the **stale banner** instead of auto-redrawing (unlike the fast `dual_plot`/QC-correlation-heatmap pages, which recolour live). **Cluster** and **show-dendrogram** are separate controls (clustering on by default — the P7d k-means hook; dendrograms a 3-state Off/Auto/On radio, Auto hides above its threshold). Colour ramp: z-score → **divergent RdBu centred at 0** (symmetric limits via `expr_symmetric_limits()`), raw → sequential viridis; source is a **Custom ramp** (default, the extracted `continuous_palette_*` control) or the assay's **Palette-page** config (raw only). Render button + stale banner sit **above the plot**; the sidebar has **Collapse/Expand all** accordion buttons and a **Plot size** panel (in-app height px + width % only, not applied to a saved image). `mod_plot_subset` "Showing:". The shared heatmap-controller refactor + advanced options (row k-means = P7d) are later.
7. **Palette** (`mod_palette.R`) — project-wide colour configuration; the **single source of truth** for discrete colour mappings. The user opts in per metadata column (empty by default), chooses a **base palette** from one `selectInput` with `<optgroup>`s (Custom / Qualitative / Brewer: Qualitative / Brewer: Sequential / Brewer: Divergent / viridis; option labels drop the `<pkg>:` prefix, values stay resolvable), and may hand-edit individual level colours (which flips the palette to **Custom palette**, so the selector never implies the named palette was modified). Per-column config is `list(name, colors)` where `colors` is the full level→hex map (everything else is inferred from `name`); it lives in `state$palette` (a UI preference, untouched by load/reset, no `data_version`). **`palette_helpers.R`** holds the low-level colour *builders* — `palette_discrete()` / `palette_gradientn()` / `palette_colorramp2()` / `palette_resolve_range()`; the **consumer-facing** resolution (turning a config + the data into a ggplot scale or a ComplexHeatmap annotation) now goes through **`aes_helpers.R`** (`aes_resolve()` → `aes_ggplot_scale()` / `aes_heatmap_col()` / `aes_annotation()`, see the `shiny-plot-aesthetics` skill), so plots and heatmap annotations agree across pages. `palette_colors(name, n)` infers the source from the name and enumerates from `viridisLite`/`RColorBrewer` (optional Suggests; graceful Okabe-Ito fallback); `palette_choices()` builds the grouped `selectInput` choices. **All resolved colours are normalized to 6-digit hex** (`norm_color()`, which also accepts R names like `gray50` / CSS names) — important because `viridis()` returns 8-digit alpha hex and a colour-picker echoes 6-digit, so without normalization the equality check would mis-fire and spuriously flip the selector to Custom. The page follows the **3-level layout** (like Input/QC): a `navset_card_tab` of **Customize** / **Config** / **Preview**. Customize holds the customizable pills (Sample / Feature / Assay / Other), each with its own `layout_sidebar` (the QC pattern). The panel machinery is **generalized over four domains** (`colData`/`rowData`/`assays`/`other`), each item **discrete *or* continuous** — colData/rowData by column type; assays continuous; `other` = the session/derived colour attributes (discrete suggested-removal + removal-pool maps, the per-sample QC-metric continuous ramps) plus the app-internal correlation ramp — **the customizable `other` set is `aes_other_palette_items()` (in `aes_helpers.R`) + `correlation` + `DEG`**, mostly the items the shared resolver reads (`correlation` and `DEG` are Palette-only — `DEG` is per-*feature*, read directly by the DE plots). **Project standard:** every attribute selectable in a colour/annotation control has a palette-config slot (`colData`/`assays`/`other`) editable here, and the shared **`aes_helpers.R`** catalog/resolver (`aes_resolve()`/`aes_ggplot_scale()`/`aes_heatmap_col()`/`aes_annotation()`) is the single place PCA + the QC plots + the heatmap annotations turn an attribute into per-sample values + colours (reading those configs). **Continuous** mappings are `list(name, min, max, custom, reverse)`: a sequential/divergent/viridis palette (or a **Custom ramp** of up to 5 colours) + anchors (a number, a `p<pct>` percentile, or blank = data range) + a **reverse** flag, resolved against the real data **at the consumer** via `palette_resolve_range()` → `palette_colorramp2()` (ComplexHeatmap) / `palette_gradientn()` (ggplot `scale_*_gradientn` pieces). The continuous panel's **widget cluster** (base-palette select + anchors + reverse + custom stops + Edit-palette/Reset) is also extracted as the standalone **`continuous_palette_*`** control (`mod_continuous_palette.R`) returning the same `list(name, min, max, custom, reverse)` — so any plot that needs its **own** local ramp (the P7c Expression heatmap; the P8 shared heatmap controller) reuses it instead of re-building the widgets. Each accordion title carries **Type** (Discrete/Continuous) + data-**Class** badges. **High-cardinality guard:** discrete attributes above `getOption("ddsdashboard.palette_max_levels", 50)` are hidden from the Add control (with an "Affected fields" note); between `…palette_warn_levels` (10) and the cap, adding shows a confirm modal. **Presets:** `other/removal_status` seeds the QC green/amber/red; `other/__pool__` seeds grey/red (kept / in pool); `other/correlation` seeds reversed RdBu anchored to [-1, 1]; QC-metric `other/__qc__*` items default to viridis; `other/DEG` (3 fixed levels up/down/no_change) seeds **DEG: Pink-Blue** and — uniquely — offers a **curated "DEG palette" group** (`deg_palette_choices()`: Pink-Blue / Orange-Purple / Red-Blue / Coral-Teal) *only* for the DEG item (the generic N-level palettes are meaningless for 3 semantic levels), resolved by a `"DEG:"` branch in `palette_colors()`, kept out of `palette_type_names()` but still shown in the Preview catalogue. Wired consumers (all via `aes_helpers.R`): the QC plots + PCA colour/shape, and the correlation-heatmap annotation tracks + value ramp (`cor_config`). The removal-status colours resolve via `removal_status_colors()` (in `filter_helpers.R`; the old `removal_palette()` closure was removed); `qc_annotation_colors()` remains the fallback for plain `colData` heatmap annotation. `assays` configs drive PCA gene colouring; `rowData` configs are stored for the P7 Expression heatmap. **Preview** is sidebar-free: catalogue swatches grouped (qualitative as discrete blocks, sequential/divergent/viridis as ramps) + its own Collapse all. (Collapse calls `accordion_panel_close()` with the *unnamespaced* id — `sendInputMessage` already namespaces.) Per-item observers read levels **live** and use `ignoreInit`; ids are `<prefix>_<domain>__<item>`. A **register-on-visible** `observe(struct())` wires any config item present in `dom_items()` (so an imported / later-added column's pickers work). `thematic` still owns fonts/fg/bg and the *default* scale when an item has no config. **Config tab** (2nd): `palette_to_json()`/`palette_from_json()` round-trip `state$palette` as a versioned, faithful mirror (kind inferred from keys; discrete `colors` → `{level: hex}` object, `custom` → array) — **selective export** (one `checkboxGroupInput` per non-empty domain driving an `export_palette()` reactive that feeds both a live JSON preview and the `ddsdashboard-palette-<date>.json` download; the selector keys off `struct()` so colour edits don't reset the selection) + **import** via one Replace/Merge/Cancel modal. `reconcile_palette()` keeps only mappings valid for the **current dataset** (drops absent columns / kind-mismatch / over-cap, trims phantom discrete levels, drops an all-phantom Custom mapping) at import (trim) **and on every dataset switch** (no trim — so routine sample/feature edits, which don't change the column set, drop nothing); a known palette whose colours diverge gets a Keep-colours/Force-palette conflict radio. (Note the discrete pin observer must look up `cols[[lev]]` safely — an atomic `[[` on a name absent from the stored colours *errors*, which froze the page before the reconcile + safe-lookup fixes.) ## Conventions that must hold

- **`rowData(dds)$feature_class`** is **always present** (factor: `endogenous` default / `spike_in` / `exogenous`). Spike-in/exogenous are **flagged, never dropped** — excluded from size-factor estimation (`controlGenes = which(feature_class == "endogenous")`) and variable-gene selection, but still in the object so their expression stays plottable.
- **`rowData(dds)$feature_length`** holds effective length (bases) when known; it may come from a *different* feature type than the rows (e.g. exonic length for gene rows) — never assume it equals the row's genomic span. GTF-derived length uses the direct `rtracklayer` union-exon route (not a `TxDb`). TPM/FPKM degrade to CPM when absent.
- **`logcounts`** assay is added automatically on load; **CPM** may be auto-added too. All count-derived assays are **recomputed whenever features change** (drives `data_version`).
- **DE result columns** (every results table carries both LFC variants):
  - `log2FoldChange` (standard) and `log2FoldChange_shrunk` (`lfcShrink`).
  - `sig`/`DEG` and `sig_shrunk`/`DEG_shrunk` — `sig` rule: `!is.na(padj) & padj < 0.05 & abs(<chosen LFC>) >= log2(2)`; `padj` is shared. `DEG` factor levels `up`/`down`/`no_change`. Thresholds user-adjustable; the shrinkage toggle selects which column set drives plots.
- **Design term order — variable of interest LAST:** the shared design builder writes **covariates first, the variable of interest last** (`~ rep + batch + condition`, DESeq2 results-name style). Order doesn't change correctness (results are extracted via `contrast=`), but it is the app standard, applied when a design is (re)built and read back. **`primary_design_var(dds)`** (`de_helpers.R`) is the single resolver for "which design variable is primary" (the **last** design term; `NA` for `~ 1`); every default group/colour/contrast picker reads it (DE contrast var, PCA/Expression/Size-factors-per-sample colour, QC within-group group) so they agree — **don't hand-roll `all.vars(design)[1]`.**
- **Feature lookup & adaptive labels:** `lookup_feature()` resolves a query against `rowData`, defaulting to `<feature_type>_name` (e.g. `gene_name`), falling back to `rownames`. The feature unit (gene/transcript/…) is detected and relabeled dynamically; prompt when unknown; let the user correct it. **ID-match convention (1:1 vs 1:many):** `lookup_feature()` is **1:1, first-match** (`match(q, v, incomparables = NA)` — one id per query, the first when a name is shared, NA never matches an NA column) — the right contract for **annotation** (each row unique → one label; see the `annotation` skill). **Gene-set building deliberately differs** — the Gene Sets import (`mod_geneset.R`) keeps **all** matching ids for a name (a set is a collection; keep-all default, toggleable to first-only) via its own 1:many resolver; rownames matching stays 1:1. Don't reuse `lookup_feature()` for gene-set matching.
- **Heading hierarchy in the UI:** `h1` = app title (navbar); `h2` = top-level `nav_panel` labels ("Input", "QC", …); `h3` = Input sub-tab labels ("Dataset", "Sample", "Feature", "Assay", "Size factors") and `card_header` titles on non-Input pages; `h4` = `card_header` titles inside Input sub-tabs and `navset_card_pill` titles. All are visually scaled with `fs-*` to fit their context. Tab/pill navigation labels (plain strings in `nav_panel`) are not headings.
- **Read-only tables render via `dt_table()`** (`R/utils_table.R`) — the standard DT wrapper giving per-column filters, a rows-per-page selector, a search box, and horizontal scroll. Don't hand-roll `DT::datatable()` options for display tables; extend via its `options=`/`...` passthrough. The editable metadata editor (`mod_meta_editor`) is the deliberate exception (it has its own editable/selection config).
- **Tooltips use `bslib::tooltip()`** (Bootstrap 5), never the native `title=` HTML attribute — so hover styling and delay stay consistent app-wide (e.g. the `mod_meta_editor` reset buttons, the Filtering Auto buttons).
- **Colour / group / annotation by a per-sample attribute goes through the shared catalog + resolver** (`R/aes_helpers.R`) — **never hand-roll colour scales or per-attribute `if/else` in a module.** `aes_catalog(state)` lists the available attributes (colData / gene / QC metrics / removal / pool / spike-in, gated to the dataset); `aes_choices()`/`group_field_choices()` build the grouped selector (General → This session → Spike-in → Data metadata); `aes_resolve(state, key, samples)` returns per-sample `values` + colours (or `NULL` when not ready → caller `validate`/skips); `aes_ggplot_scale()` / `aes_heatmap_col()` / `aes_annotation()` adapt that to a ggplot scale or a ComplexHeatmap annotation. **Project standard:** every selectable attribute has a palette-config slot (`colData`/`assays`/`other`, the descriptor's `loc`) and is editable on the Palette page; the customizable "Other" set is `aes_other_palette_items()`. PCA, the QC plots, and the correlation-heatmap annotation are wired this way (and future t-SNE/UMAP/DE pages reuse it). Display aesthetics are cheap re-plots — they must **not** enter the `state_derive` key or the `deferred()` `sig`. **The one documented exception is a slow static `ComplexHeatmap`** (the P7c Expression heatmap): a full heatmap redraw is *not* cheap, so there its colour ramp + annotation colours are **snapshotted and gated behind Render** (in the `sig`), trading live recolour for a stale banner — see the Expression Heatmap description. The fast `dual_plot`/PCA/QC-plot pages and the QC *correlation* heatmap keep colours live. **Full reference: the `shiny-plot-aesthetics` skill.**
- **Plot-visualization tabs embed the view-only "Showing:" subset control** (`R/mod_plot_subset.R`): `plot_subset_ui(ns, <suffix>)` in each plot sidebar + one `plot_subset_server(input, output, session, state, suffixes)` call returning a `showing_samples` reactive the plots filter their data by (and the deferred `sig`s depend on). It's display-only — never mutates the `dds` or bumps `data_version`. This is the standard for any page with per-sample plots (QC today; PCA/DE later).
- **Deferred rendering:** plots can be slow. Gate re-rendering behind an explicit "apply / render" button (`bindEvent()` / `eventReactive()`), not live reactivity on every input. The **Auto-render** control that pairs with a Render button is a **`bslib::input_switch`** (a toggle, not a `checkboxInput`) app-wide.
- **ggplot↔plotly engine toggle** (`mod_qc.R`): a global status-bar switch sets `state$plot_interactive` (default off = static ggplot). Toggleable ggplot plots use the **dual-output** pattern — a `gg(interactive)` builder fn feeds `dual_plot("<id>", gg, n_elements)`, which wires a static `plotOutput` (always `gg(FALSE)`) + an interactive `plotlyOutput` (`ggplotly`, `gg(TRUE)`) behind a `uiOutput` container; the UI placeholder is `.qc_dual_plot(ns("<id>_container"))`. Interactivity needs `use_plotly_base()` (toggle ON **and** `requireNamespace("plotly")` — a `Suggests`, graceful fallback) and is then gated **per plot on an element budget**: `n_elements()` (≈ rows of the plotted data, the real `ggplotly` cost driver) vs `.plotly_max_elements()` = `getOption("ddsdashboard.plotly_max_elements", 5000L)` — tunable without a settings page. Over budget → static + a **sticky per-plot "Render interactive anyway"** override (`forced` reactiveVal, reset on `data_version` change or toggle flip). Hover comes from a `text` aes the builders add only when `interactive = TRUE`; the plotly path muffles the residual aesthetic warning (`.muffle_unknown_aes`) and catches `validate()`/`req()` into a message figure. **Not** for ComplexHeatmap / non-ggplot outputs. Reuse for future plot pages (PCA/DE).
- **Caching is the `derived` store** (see State model): VST/PCA/DESeq fit are version-stamped and invalidated when `data_version` changes. `bindCache()` or keyed entries both work.

## Workflow

- Build the app in **phases** (bulk-first). **The detailed phase plan + current status lives in [dev_ref/roadmap.md](dev_ref/roadmap.md) — consult/update it as the source of truth** (don't duplicate the full list here). At a glance: P1 (skeleton), P2 (annotation + normalization + theming), **P3a–g** (QC, filtering, edit-history controls, ERCC dose-response, spike-in QC filtering, ggplot↔plotly engine toggle, Palette page incl. JSON import/export) **P4a** (single-panel PCA + the colour/annotation resolver epic) and **P5** (DESeq2 DE — design/contrast builder, MA/volcano/direct plots, results table, `shiny-de-analysis` skill) are **done**; **P6 (Gene Sets page) is next**, then P7 (Expression — Single genes + Gene sets heatmap), P8 (Visualization enhancements: P3g-d factor mgmt, P4b multi-panel PCA, P4c t-SNE/UMAP, shared heatmap controllers, + wishlist), P9 (Export & reproducibility: R script/Quarto report, plot export, DE tables, gene sets), P10 (single-cell, lowest priority).
- **Build mock-`dds` fixtures early** (`data-raw/`, used by `tests/`) so every phase has data to test against.
- **After each code-editing round (any `R/` change), run the test suite before finishing** — `mamba run -n rnaseq_dashboard Rscript -e 'devtools::test()'` — and report the pass/fail count. Run it once per logical round (not after every single file), and always before a commit; skip it for docs-only changes.
- **Commit after each meaningful, self-contained change** with a descriptive message.
- **Version & releases — propose, don't self-serve.** Scheme is `0.MINOR.PATCH` (pre-1.0): a **completed roadmap phase** bumps the **minor** (`v0.1.0` = Phases 1–3, `v0.2.0` = Phase 4/PCA, `v0.3.0` = Phase 5/DE, …); a **key developmental step within a phase** (typically a meaningful sub-PR) bumps the **patch** (`v0.3.1`, `v0.3.2`, … then phase-complete rolls to `v0.4.0`). `v1.0.0` = the bulk workflow feature-complete through Export/reproducibility (Phase 9); single-cell (Phase 10) is post-1.0. Keep the **`DESCRIPTION` `Version` and the git tag / GitHub release in lockstep** (tag = `vX.Y.Z`). At a phase completion — or a key sub-step — **propose** the bump + tag; don't create a tag/release unsolicited (it's outward-facing — wait for the user's go-ahead). Release flow when approved: bump `DESCRIPTION` to `X.Y.Z` → tag `vX.Y.Z` (+ GitHub release, optional `NEWS.md` entry) → bump `DESCRIPTION` to `X.Y.Z.9000` to resume dev.
- **Keep docs in sync with the standards they describe — PR-based, not just the roadmap.** Doc-sync is **per-PR, not deferred to a phase close-out**: when a PR touches a **project standard or convention** — a module promoted to project-wide use, a convention moved/renamed, a new shared helper others must route through (e.g. the `aes_helpers` resolver), a **divergent contract** (e.g. the gene-set 1:many resolver vs annotation's 1:1 `lookup_feature`), a dependency-policy or state-model change — update the standards docs (`CLAUDE.md`, the relevant `.claude/skills/*`, the `.claude/agents/*`) **in that same PR**. Updating [dev_ref/roadmap.md](dev_ref/roadmap.md) status is *not* enough when the *contract* changed, and the docs must not drift by a whole phase before catching up. Routine feature work that fits existing conventions needs only the roadmap update.

Spawn it only for genuine standard changes (the cold-start re-read isn't worth it for a one-line nudge — do those inline); give it the concrete delta (what changed, which files) so it isn't re-deriving. Keep judgment-heavy nuance (the *why* behind a load-bearing rule) in the brief — don't expect a small model to infer it.
